# Passkey (WebAuthn) Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make passkey sign-in and registration work on any site in Evo by carrying the Apple-restricted `com.apple.developer.web-browser.public-key-credential` entitlement on a signed build — WKWebView then performs WebAuthn natively and routes to the active system provider (1Password or iCloud Keychain).

**Architecture:** There is essentially no application code. WKWebView surfaces WebAuthn automatically once the app is signed with a provisioning profile that authorizes the entitlement. The work is (1) declare the entitlement in `project.yml`, (2) switch the Debug build from unsigned to signed so it can carry the entitlement, (3) validate on real passkey sites. Steps split into **verifiable-now** (the inert entitlement key) and **GATED** (everything requiring the signed build, which cannot be done until Apple grants the managed capability).

**Tech Stack:** XcodeGen (`project.yml` → generated `Evo.xcodeproj` + `evo/Info/evo.entitlements`), Xcode code signing / provisioning profiles, WKWebView native WebAuthn (macOS 13.3+), Apple Developer "Capability Requests".

## Global Constraints

- Add **exactly one** entitlement key: `com.apple.developer.web-browser.public-key-credential: true`. Do **NOT** add the parent `com.apple.developer.web-browser` (default-browser registration — separate, unneeded feature).
- Debug build keeps `ENABLE_HARDENED_RUNTIME: NO` (the embedded Go `evo-op-helper` sidecar and its `dlopen` of the 1Password IPC dylib depend on this).
- **No Swift, JavaScript, or Go source changes.** If a task appears to need one, stop — the design is wrong.
- App ID / bundle identifier that must carry the capability: `com.skproductions.evobrowser`.
- WebAuthn requires macOS 13.3+; Evo targets macOS 15.0 — satisfied, no deployment-target change.
- `evo/Info/evo.entitlements` is generated by `xcodegen` from `project.yml` **and** committed. After any `project.yml` entitlements edit: run `xcodegen`, then commit `project.yml` and the regenerated `evo/Info/evo.entitlements` together.
- **The signed-build tasks (4–6) are BLOCKED until Apple grants the capability (Task 3).** Do not attempt them before the grant — the provisioning profile cannot include an unassigned managed entitlement, so the build will fail.

### Task 4: [GATED on grant] Switch the Debug build to signed with the entitlement

Once the App ID carries the capability, sign the Debug build so the entitlement takes effect. The embedded Go sidecar must also be signed for the app's signature to stay valid.

**Files:**
- Modify: `project.yml` (Debug config under `targets.evo.settings.configs.Debug`; and the `evo-op-helper` copyFiles phase)
- Modify: `scripts/xcbuild-debug.sh`
- Regenerate: `Evo.xcodeproj` (via `xcodegen`)

**Interfaces:**
- Consumes: the entitlement declared in Task 1.
- Produces: a signed Debug `Evo.app` whose code signature includes the passkey entitlement — consumed by Task 6 validation.

- [ ] **Step 1: Discover your signing identity and team ID (real values, not placeholders)**

Run: `security find-identity -v -p codesigning`
Note the **"Apple Development: …"** identity and its 10-character Team ID (the string in parentheses, e.g. `(ABCDE12345)`). Use these concrete values in the steps below wherever they say *your Team ID* / *your Apple Development identity*.

- [ ] **Step 2: Regenerate the development provisioning profile (portal)**

In the Apple Developer portal, regenerate (or create) a **Mac Development** provisioning profile for App ID `com.skproductions.evobrowser` now that the capability is granted, so the profile authorizes the entitlement. Download it, or rely on Xcode automatic provisioning (Step 3 tries Automatic first).

- [ ] **Step 3: Flip the Debug config to sign (try Automatic first)**

In `project.yml`, change the `Debug` config from:

```yaml
        Debug:
          ASSETCATALOG_COMPILER_APPICON_NAME: EvoIconDev
          CODE_SIGN_ENTITLEMENTS: evo/Info/evo.entitlements
          CODE_SIGN_STYLE: Automatic
          EMIT_FRONTEND_COMMAND_LINES: YES
          OTHER_LDFLAGS: "-Xlinker -interposable"
          ENABLE_HARDENED_RUNTIME: NO
```

to (add `DEVELOPMENT_TEAM` and a development signing identity; keep hardened runtime off):

```yaml
        Debug:
          ASSETCATALOG_COMPILER_APPICON_NAME: EvoIconDev
          CODE_SIGN_ENTITLEMENTS: evo/Info/evo.entitlements
          CODE_SIGN_STYLE: Automatic
          CODE_SIGN_IDENTITY: "Apple Development"
          DEVELOPMENT_TEAM: "YOUR_TEAM_ID"   # the 10-char Team ID from Step 1
          EMIT_FRONTEND_COMMAND_LINES: YES
          OTHER_LDFLAGS: "-Xlinker -interposable"
          ENABLE_HARDENED_RUNTIME: NO
```

Replace `YOUR_TEAM_ID` with the real Team ID from Step 1.

- [ ] **Step 4: Sign the embedded sidecar on copy**

Find the copyFiles phase that embeds the helper: `grep -n "evo-op-helper" project.yml`. In that copyFiles entry, ensure the file is signed when copied by adding `codeSignOnCopy: true` to its attributes (XcodeGen supports this on `copyFiles` items). Example shape — match the existing entry's structure, adding the flag:

```yaml
        - path: tools/op-helper/bin/evo-op-helper
          buildPhase:
            copyFiles:
              destination: executables
          codeSignOnCopy: true
```

This prevents an invalid deep signature from an unsigned nested Mach-O.

- [ ] **Step 5: Make the debug build script sign instead of forcing unsigned**

Replace `scripts/xcbuild-debug.sh` with:

```bash
#!/bin/bash
set -o pipefail && xcodebuild build \
  -scheme evo \
  -destination "platform=macOS" \
  -configuration Debug \
  -allowProvisioningUpdates | xcbeautify
```

(The unsigned overrides `CODE_SIGN_IDENTITY="" CODE_SIGNING_REQUIRED=NO` are removed so the project's signing settings apply; `-allowProvisioningUpdates` lets Xcode fetch/refresh the profile.)

- [ ] **Step 6: Regenerate and build signed**

Run: `xcodegen && ./scripts/xcbuild-debug.sh`
Expected: `** BUILD SUCCEEDED **`, signed with your Apple Development identity.
If it fails with an entitlement/provisioning error under Automatic signing: switch to Manual — set `CODE_SIGN_STYLE: Manual` and add `PROVISIONING_PROFILE_SPECIFIER: "<the profile name from Step 2>"` in the Debug config, then rerun. (Managed entitlements sometimes do not auto-provision.)

- [ ] **Step 7: Verify the entitlement is actually embedded in the signature**

Run:

Expected: a line showing `com.apple.developer.web-browser.public-key-credential` with value `true`. If absent, the entitlement did not take — do not proceed; recheck the profile/signing.

- [ ] **Step 8: Verify the embedded sidecar is signed**

Run: `codesign -dv "$APP/Contents/MacOS/evo-op-helper" 2>&1 | grep -i 'Signature\|Authority' | head`
Expected: shows a valid signature (not "code object is not signed at all").

### Task 6: [GATED on grant] Passkey validation UAT

Human-in-the-loop validation on the signed Debug build. There are no automated tests for this feature — the behavior is OS-brokered and the project's test flow runs unsigned (so a runtime entitlement assertion would fail there). Verification is signature inspection (Tasks 4–5) plus these manual cases. **User in UAT = Sam.**

**Preconditions:**
- Signed Debug build from Task 4 is running (`codesign` confirmed the entitlement).
- 1Password 8 desktop app is unlocked and enabled as a system passkey/AutoFill provider (System Settings → General → AutoFill & Passwords → enable 1Password), so the OS offers 1Password passkeys.

- [ ] **UAT-1 — Registration (`create`) via 1Password:** Go to `https://webauthn.io`. Register a new passkey. Expect the macOS passkey sheet to appear and offer 1Password; complete it. Expect success and the new passkey to appear in 1Password.
- [ ] **UAT-2 — Authentication (`get`) via 1Password:** On `https://webauthn.io`, authenticate with the passkey just created. Expect the system sheet → 1Password → successful sign-in.
- [ ] **UAT-3 — Real site, existing passkey:** On a site where Sam already has a passkey in 1Password (e.g. GitHub), start passkey sign-in. Expect the ceremony to complete via 1Password. (This is the exact flow that failed pre-entitlement.)
- [ ] **UAT-4 — iCloud Keychain provider (sanity):** With 1Password disabled as provider (or choosing "iCloud Keychain" in the sheet), create/use a passkey on `webauthn.io`. Expect the system sheet to route to iCloud Keychain. Confirms it is genuinely system-brokered, not 1Password-specific.
- [ ] **UAT-5 — Sidecar regression (fill still works):** On a normal login page, focus the username/password field. Expect the 1Password sidecar to spawn (Touch ID once) and the fill overlay to appear and fill username/password + TOTP as before. Confirms signing the Debug build + the helper did not break the form-fill integration.
- [ ] **UAT-6 — No regressions in normal browsing:** Confirm hot reload (Inject), private windows, and general browsing behave normally on the signed build.
